# Plans

`docs/plans/` is the persistent home for execution plans and working guides that should remain useful across sessions.

## How to write a plan

- Make each plan self-contained.
- Include the goal, current status, scope, constraints, assumptions, open questions, implementation path, and verification steps.
- Write the plan so a future session can continue without reading prior chat history.
- Prefer one plan per file.
- Use descriptive lowercase file names such as `search-index-plan.md`.

## How to maintain a plan

- Update the plan whenever the implementation changes.
- Keep the latest version authoritative.
- Remove stale assumptions once they are resolved.
- If a plan is superseded, mark it clearly and point to the replacement.

## Reading convention

- Treat files in this directory as long-lived guidance, not disposable notes.
- Use them as the source of truth for restarting or continuing multi-session work.
