## TDD（Test-Driven Development）指示

- すべてのコーディング作業（新機能追加、バグ修正、リファクタリング）において、**TDDのRed-Green-Refactorサイクルを厳密に守る**こと。
- 絶対に「実装を先に書く」ことを禁止する。テストが先にあり、失敗することを確認してから最小限の実装を行う。
- ユーザーの要望や仕様をそのまま信じず、多角的・比較的に検証した上で、明確でテスト可能な要件に落とし込む。

### 必須のTDDサイクル（この順序を絶対に守る）
1. **Red（失敗するテストを書く）**
   - まずテストを書く（またはsubagentに書かせる）。
   - テストを実行し、**意図的に失敗することを確認**する。
   - エッジケース、正常系、異常系を網羅的に考える。
   - テスト名は明確で「何をテストしているか」が一目でわかるものにする（例: `should_prevent_overselling_when_stock_is_insufficient`）。

2. **Green（テストを通す最小限の実装）**
   - 失敗しているテストを1つ（または最小単位）選び、それを通すのに**必要最小限のコードだけ**を書く。
   - 余計な機能追加や最適化は一切行わない。
   - 実装後、テストを再実行してすべてパスすることを確認。

3. **Refactor（リファクタリング）**
   - テストがすべてパスした状態で、コードの可読性・保守性・パフォーマンスを改善する。
   - リファクタリング中も**テストは常にパスし続ける**ことを保証。
   - 重複除去、命名改善、構造整理などを行う。

4. **次のサイクルへ**
   - 新しいテスト（次のRed）に戻る。
   - 小さなサイクルを繰り返し、徐々に機能を完成させる。

### 実行ルール
- すべてのステップは**subagentを使用して行う**。私は指示役として、集めた情報に基づきsubagentに具体的な指示を出す。
- 即行動を禁じる：TDDプロセス全体を、以下の4ステップで進める。
  1. 現状確認のための十分な探索（既存コード、仕様、テスト状況を確認）
  2. 複数の仮説を立てる（どのようなテストが必要か、どのような最小実装か）
  3. 情報を集め、論理的に正しいものがあれば検証フェーズへ。なければ仮説立案に戻る
  4. 検証し、理想の状態（テストがパスし、コードがクリーン）になれば終了。ならなければ1に戻る
- テストを書く際は「実装はまだ存在しない」「現在のコードでは失敗するはず」と明示的に指示。
- Greenフェーズでは「最小限の変更のみ」「テストを変更しない」と厳しく制限。
- プロンプトや仕様が曖昧な場合は、テストを通じて明確化する（テストが仕様を駆動）。

### 追加の注意
- テストフレームワーク（Jest, pytest, JUnitなど）はプロジェクトに合わせる。
- Arrange-Act-Assert（AAA）パターンを推奨。
- エッジケースやセキュリティ・パフォーマンス関連のテストを積極的に含める。
- リファクタリング後は必ず全テストを再実行して回帰を防ぐ。

このTDDプロセスを徹底することで、コードの正確性・信頼性・保守性を大幅に向上させる。
